3. **주·일 학습계획** — 교시×요일 그리드 + 교재/학습내용 표시. 학생은 조회+입력분, 관리자 편집(교시·시간·주말 의무자습·선택목록) (→ 8.8)

5. **영단어 시험(디랩 콘텐츠)** — 앱 응시 + 결과 링크(Daily Report 참고링크) (→ 8.9)
6. **메시지 — 가족 채팅방** 추가(학부모+학생+관리자 공동) (→ 8.10)

## 4. 공통 설계 원칙

### 4.1 API 소비 규약 (Dlab통합관리프로그램(구 DSA) 공통 응답 형식 준수)
```dart
// 공통 응답: { success, data, meta, error }
class ApiResult<T> {
  final bool success;
  final T? data;
  final PageMeta? meta;
  final ApiError? error; // { code, message, details }
}
```
- 모든 목록 호출은 서버 규약대로 `year`, `branchId`를 포함(로그인 컨텍스트에서 자동 주입).
- 페이지네이션: `page`, `size`, `sort` 쿼리.

### 4.2 Dio 인터셉터
```dart
dio.interceptors.addAll([
  AuthInterceptor(),   // Authorization: Bearer {accessToken}, 401 시 refresh
  RetryInterceptor(),  // 네트워크/5xx 지수백오프 재시도
  ErrorMapInterceptor(),// ApiError.code → Failure 매핑
  LoggingInterceptor(),// prod 비활성
]);
```
- **401 → Refresh Token으로 갱신 → 원요청 재시도**. Refresh 실패 시 로그아웃.
- 멱등하지 않은 POST(결제 등)는 재시도 제외 + **Idempotency-Key** 헤더.

### 4.3 상태·서버데이터
- 화면별 `AsyncNotifier`로 로딩/에러/데이터 3-state 통일. 공통 `AsyncView` 위젯으로 렌더.
- 낙관적 업데이트는 승인/토글류에만 제한적으로.

### 4.4 멀티지점(branch) 스코프
- 로그인 응답에 `branchId` 포함 → 전 요청 컨텍스트 주입. Daily Report 랭킹의 "전체 지점" 뷰는 서버가 집계.

### 4.5 오프라인/재시도
- 급식 결제·사유 제출 등 중요 쓰기는 **로컬 큐(drift)** 에 적재 후 재전송(중복 방지 Idempotency-Key). 조회는 last-known 캐시 후 갱신.

## 5. 인증 & 온보딩

### 5.1 계정 유형
- **학생**: 앱 회원가입(자체 계정, Supabase Auth) → JWT.
- **학부모**: **카카오 로그인** → 학생 연락처 입력 → 서버가 학생과 매핑(최대 2인).
- **관리자**: 직원 계정(Dlab통합관리프로그램(구 DSA)와 동일 사용자 풀, Role=TEACHER/STAFF 등)으로 앱 로그인.

### 5.2 🆕 학생 온보딩 4단계 (상태 머신)
```
REGISTERED → OT_DONE → PARENT_LINKED → SCHEDULE_SET → ACTIVE
   회원가입     OT(대면)    학부모 인증(알림톡)   정기일정 등록
```
- 🟡 **I: OT(대면) 게이트 여부** — OT 완료 전 기능 제한 범위 확정 필요. 초안: OT 전에는 홈/공지만, 출결·급식 등은 `ACTIVE`부터.
- 각 단계는 서버 `student.onboarding_status`로 관리, 앱은 상태에 따라 라우팅.

### 5.3 앱이 호출하는 인증 API (계약)
```
POST /api/v1/auth/login                 # 학생 로그인 → {access, refresh, user}
POST /api/v1/auth/refresh               # 토큰 갱신
POST /api/v1/auth/parent/kakao          # 학부모 카카오 인증
POST /api/v1/onboarding/link-parent     # 학생 연락처로 학부모 연동
GET  /api/v1/me                         # 프로필·role·branch·onboarding_status
POST /api/v1/push/token                 # FCM 토큰 등록/갱신
```

### 5.4 토큰 보관
- Access/Refresh → `flutter_secure_storage`(Keychain/Keystore). 메모리 캐시 병행.
- 앱 백그라운드 진입 시 민감화면 마스킹(11장).

## 6. 알림 설계

### 6.1 🔧 채널 이중화 (확정)
| 채널 | 용도 | 처리 주체 |
|---|---|---|
| **카카오 알림톡** | 출결·수납 등 **트랜잭션성** + 심사 통과 항목 | 서버(Kakao API) |
| **FCM Push** | **Daily Report**·앱 내 알림·승인 요청 대부분 | 서버 → FCM → 앱 |

> 🔧 **Daily Report는 FCM 우선.** 사유: 알림톡 템플릿 심사 리스크. 알림톡 병행 여부는 심사 통과 결과로 확정(I-14).

### 6.2 앱 알림 매핑 (7/10 승인 라우팅 반영)
| 알림 | 수신자 | 채널 |
|---|---|---|
| 입·퇴실 | 학부모 | FCM |
| 사유·정기일정 승인 요청 | 🆕 학부모 **또는 선생님** (10장 라우팅) | FCM |
| 방화벽 해제 승인 요청 | 🆕 학부모(유지 여부 미결) | 알림톡 |
| 벌점·졸음·전자기기 적발 | 학부모 | FCM |
| Daily Report | 학생·학부모 | 🔧 **FCM** (알림톡 병행 TBD) |
| 앱 메시지·공지 | 학생·학부모 | FCM |

### 6.3 FCM 구현 포인트

### 7.1 출결 — 동적 QR (앱 → 키오스크)

### 8.6 Daily Report (🔧 FCM · 🖼️ 0710 대시보드 확정)
> 🖼️ **클라이언트 레이아웃(0710) 반영 — 대시보드형으로 구성 확정.** 목적: 하루 사용을 직관적으로 보여주고 학생 셀프 피드백 유도.
- `GET /api/v1/daily-reports/{date}` · `GET /api/v1/daily-reports/monthly?month=YYYY-MM`(월 달력 뷰).

**구성(대시보드)**
| 영역 | 내용 |
|---|---|
| 헤더 | 지점·이름·재수 구분(N수)·좌석·학번 + **학생의 다짐 한마디**(셀프 피드백, 학생 1줄 직접 입력) → `PATCH /api/v1/daily-reports/{date}/self-feedback` |
| 순공 랭킹 | **전체 1등 + 지점 1등 동시 표시**, 순공시간 누적(예: 287h 30m) |
| 출석 요약 | **도넛 %** + 출석/지각/결석 건수 |
| 데일리 테스트 | 당월 **응시 횟수** + 결과 그래프(이번/오늘/누적 평균, fl_chart) |
| 월 달력 | 일자별 **순공시간·출결색(정상/지각·조퇴)·질의응답(질)·데일리테스트(R)** 아이콘 → 일자 탭 시 **타임라인**(입·퇴실, 외출 N분 경과) |
| 진도(플래너) | 교재·**인강 진도율** |
| 질의응답 기록 | 담임/멘토 질문 내용·**횟수** |
| 진입 버튼 | **「학습 상담 상세보기」**(→ 8.5 담임 상담: 성적 상세·학습계획 이행률) · **「사유출결/벌점 현황」**(당월 표: 날짜·벌점·사유 + 이번 달 총 벌점) |
| 리포트 진입 | **전월 리포트 / 당월 리포트** 카드(§9.1 홈 상단) |

- 매일 밤 **FCM 푸시**로 요약 전달 → 탭 시 상세.
- 🟡 순공시간 산출 정의(입퇴실/순찰/좌석없음 차감)·랭킹 집계 기준 확정 필요(I-6).

### 8.8 🆕 주·일 학습 계획 (표시/부분 입력 · 🖼️ 0710 그리드 확정)

- 🖼️ **레이아웃(0710):** 주간 뷰 = **교시(0교시~야3)×요일(월~일) 그리드**. 각 셀 = **과목 + 학습형태(수업·인강·자습) + 교재/학습 내용 텍스트**(예: `수자습 / 드릴 수1 워크북`).

## 11. 보안 (앱 관점 ISMS)

> ISMS-P 인증 자체는 개발팀 소관 아님(학원 주관). 앱은 **개발 반영 가능한 보호대책**만 책임.

| 구분 | 앱 반영 |
|---|---|
| 전송 암호화 | 전 구간 HTTPS/TLS, (옵션) 인증서 피닝 |
| 토큰 보관 | Keychain/Keystore(`flutter_secure_storage`), 메모리 최소화 |
| 재인증 | 민감 동작(결제·승인) 시 생체/PIN 옵션(`local_auth`) |
| 화면 보안 | 백그라운드 스냅샷 마스킹, 민감 화면 스크린샷 차단(옵션) |
| 개인정보 표시 | 연락처·주소 마스킹 규칙 준수(서버 마스킹 + 앱 표시) |
| 민감정보(성적) | 저장 캐시 최소화·암호화, 로그 미기록 |
| 미성년자 | **만 14세 미만 가입 시 법정대리인(보호자) 동의** 플로우 |
| 로깅 | prod 상세 로그 비활성, PII 로그 금지, Crashlytics PII 제외 |
| 딥링크 | 화이트리스트 검증(오픈 리다이렉트 방지) |
