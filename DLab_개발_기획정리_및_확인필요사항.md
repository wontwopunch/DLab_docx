## 0.1 🆕 7/10 미팅 반영 요약 (이번 버전에서 추가된 것)

빠른 파악용으로 이번에 새로 들어간/바뀐 결정을 먼저 모았습니다. 상세는 각 섹션 참조.

1. **입학예약폼 = 디멤버 사이트** — 폼 수정이 필요하며, **대성 측이 확정 후 개발팀에 전달** (→ 4.2 / S-1)
2. **앱 홈·학습 화면 표시 요소 확정** — 타임라인 + 전월 리포트 + 당월 리포트 + 교재 진도 + 데일리루틴(학생 테스트) (→ 3.14)
3. **모의고사 성적 상세 신규** — 앱은 과목별 그래프 변화 추이, **상세는 담임 상담용으로 통합관리프로그램에 표시**, **엑셀 업로드로 자동화**. 엑셀 양식 제작 여부·내년 양식 변경 대응 미결 (→ 3.9 / 4.6 / I-11)
4. **학습계획 비교그래프 확정** — 관리자 페이지에서 주간 학습계획을 **과목별(+통합사회·통합과학) × 유형(수업·인강·자습)** 비교그래프로 (→ 4.11)
5. **메시지 기능 확정** — 전체/반별/개별/지점별 공지 + 학생-담임 1:1 채팅(외부 유료 프로그램) + 행정 요청(항목 확정 후 클릭 전송) + 알림톡 병행 (→ 3.14 / 4.11)
6. **⭐ 승인 플로우 변경** — 학생 앱 신청 시 **학부모를 건너뛰고 선생님이 바로 처리**하도록 변경. 단 **항목별로 승인 주체를 분리**해야 함. **방화벽 해제만 학부모 승인 유지 여부 미결** (→ 3.15 / I-12)
7. **대성학력개발연구소(DSD) 관련** — 상담신청이 Dlab통합관리프로그램에 API 연동 중 → **해당 엔드포인트 필요**. **앱 배포 스토어 계정은 DSD 보유**. **ISMS 인증은 개발팀 소관 아님**(내부 검토 후 개발 관련만 요청 시 반영) (→ 5 / S-4 / S-5)
8. **진행 프로세스·일정 확정** — 오늘 미팅+엑셀 자료 수령 → 차주 메인 디자인·전체 메뉴·상담/성적관리 중점 산출 → 피드백 반영 최종본 (→ 10)
9. **메신저 유료 프로그램 선정 필요** — 1만 명 기준, 프론트 React·백엔드 Java, **프론트 연동은 유현님** (→ E-6 / 10)

## 0.2 🖼️ 0710 클라이언트 레이아웃 반영 요약 (디랩 Daily Report / 강지웅 과장)

`디랩_Daily_Report_0710.pdf` 요구사항 반영. 상세는 각 섹션 참조.

1. **Daily Report = 대시보드형 확정** — ① 학생 다짐 한마디(셀프 피드백, 학생 1줄) ② 순공랭킹 **전체 1등 + 지점 1등** ③ 출석 도넛% ④ 데일리테스트 횟수 ⑤ **월 달력**(일자별 순공시간·출결색·질의응답'질'·데일리테스트'R' + 타임라인) ⑥ **「학습 상담 상세보기」·「사유출결/벌점 현황」 버튼** ⑦ 교재·인강 진도율 (→ 3.10 / 3.14)
2. **상담 리포트 구조 확정** — 과목별 **이행률 별점** + **상담 항목 선택형(관리자 편집·최대 노출 개수)** + **담임 스티커** + 코멘트. Weekly ABC(데일리루틴 맞은갯수/총갯수)·수강진도 결합 (→ 4.6)
3. **주간 학습계획 = 교시×요일 그리드** — 셀=과목+학습형태(수업·인강·자습)+**교재/학습 내용**. 관리자 편집(교시·시간·주말 의무자습·선택목록). **과목/형태별 시간·비율 파이(주차)** (→ 4.11)
4. **모의고사 성적** — 6과목(국·수·영·한국사·탐1·탐2) **회차별 표**(원/표/백/등급) + 과목별 상세(단원별 정답률·전국 대비) (→ 3.9 / 4.6)
5. **질의응답 ON/OFF 분리** — ON(온라인 게시판) + **OFF(대면: 상담실 스케줄·희망 타임 예약)**. 질의응답 횟수는 Daily Report 반영 (→ 3.11)
6. **영단어 시험(디랩 콘텐츠)** — 앱 응시 + 결과 링크(Daily Report 참고링크) (→ 3.11)

## 1. 프로젝트 개요 & 목표

### 1.1 배경
- 현재 대성전산이 소유·운영하는 **Dlab통합관리프로그램(Dresung Admin)** 사용 중.
- 문제: ① 매년 초 전체 재세팅(기초데이터 수동 입력) ② 검색·정렬 부재 ③ 소유권 없음(수정 시 대성전산 의뢰) ④ 구글 시트 2개 병행(데이터 분산) ⑤ 앱 실시간 연동 구조적 불가.
- 목표: **DLab이 직접 소유하는 통합 플랫폼**(관리프로그램 + 재원생 앱) 신규 개발. 데이터 소유권을 학원 소유 DB로 이전.

### 1.2 개발 대상 (2개 산출물)
| 산출물 | 스택 | 사용자 |
|---|---|---|
| 재원생 전용 앱 | **Flutter (iOS 15+ / Android 10+)** | 학생 · 학부모(최대 2인) · 관리자(현장 순찰 등) |
| 통합 관리 프로그램 | **웹 (React)** — PC Chrome 최신, 설치 불필요 | 관리자 · 직원 · 강사 |

### 1.3 운영 주체 / 관계사
| 주체 | 역할 | 개발 시 접점 |
|---|---|---|
| 대성학력개발연구소(DSD) | D.Lab 사이트 운영 | 입학예약·성적표 PDF·교재 웹뷰·**상담신청 API**·**앱 스토어 계정** 연동 |
| 대성전산 | 기존 Dlab통합관리프로그램·수납·디멤버 소유 | 🔴 데이터 이관 · 수납 API · 키오스크 · **디멤버 입학예약폼** |
| 자이엘코리아 | Zyxel 방화벽 장비 관리 (김상원 팀장 / 010-6412-8131) | 🔴 Nebula API 권한 |
| 급식업체 | 급식 공급, PG 가맹점 명의 | 🔴 연락처·PG·환불 프로세스 |

> 🆕 **7/10:** 앱 배포용 **스토어 계정은 DSD가 보유** 중 → 신규 개발 앱 배포 시 해당 계정 사용 여부/권한 위임 방식 협의.

## 2. 시스템 아키텍처

```
[ 재원생 앱 (iOS/Android) ] + [ 키오스크 ] + [ DLab 사이트 ]
        │
        ▼  (모두 동일한 Spring Boot API 서버에 직접 연동)
[ Spring Boot API 서버 (Java) ] ──── [ Supabase (PostgreSQL, 학원 직접 소유) ]
        │
        ▼
[ 통합 관리 프로그램 (React 웹) ]
        │
        ▼  API 연동
[ Zyxel Nebula ] [ 더프리미엄모의고사 API ] [ PG사 ] [ 카카오 알림톡 ]  ( + ⚠️대성전산 API )
```

### 2.1 기술 스택 (확정)
| 구분 | 기술 |
|---|---|
| 앱 | Flutter (iOS/Android) |
| 관리 프로그램 | React (웹) |
| 백엔드 API | Spring Boot (Java) — 앱·관리 공통 서버 |
| DB | Supabase (PostgreSQL) — 학원 직접 소유 |
| 알림 | 카카오 알림톡 + FCM Push |
| 키오스크 연동 | Webhook 직접 수신 (대성전산 API 경유 제거 목표) |
| 방화벽 | Zyxel USG Flex 700H + WBE510D + WBE630S + GS1920-24HP (운영 중) |
| 성적 | 더프리미엄모의고사 API (전화번호 기반) + 🆕 엑셀 업로드 (상세/상담용) |
| 결제 | PG사 카드·가상계좌 (급식업체 명의 가맹점) |
| 1:1 메신저 | 🆕 외부 유료 메신저 (React 연동 · 1만 명 기준 선정) |

### 2.2 아키텍처 관점 개발 유의점 (초안)
- **단일 API 서버 공유**: 앱·키오스크·관리프로그램이 같은 Spring Boot 서버를 바라봄 → 인증/권한(Role) 설계를 서버 레벨에서 통일 필요.
- **멀티테넌시(다지점)**: 지점별 독립 데이터 + 상위 관리자 전 지점 조회 → 모든 주요 테이블에 지점(branch) 스코프 및 RLS/RBAC 설계 필요.
- **학번 매년 초기화**: 학번을 PK로 쓰면 안 됨 → 내부 불변 학생 ID(UUID 등) + 연도별 학번 매핑 구조 필요.
- **실시간성**: 입퇴실 알림 30초 이내, 키오스크 Webhook 즉시 반영 → 비동기 알림 큐 설계 고려.
- 🆕 **승인 라우팅 파라미터화**: 신청 항목별로 승인 주체(학부모/선생님)가 갈리므로, 승인 대상을 **항목 설정값으로 관리**하는 구조 필요 (3.15 참조).

## 3. 프로젝트 A — 재원생 앱 기능 정리

### 3.1 사용자 유형
| Role | 대상 | 주요 기능 |
|---|---|---|
| 학생 | 재원생 | 출결 확인, 사유 제출, 질의응답, 급식 신청, Daily Report |
| 학부모 | 보호자(최대 2인) | 사유 승인, 출결·벌점 조회, 방화벽 해제 승인, Daily Report |
| 관리자 | 담임·사감·직원 | 순찰 기록, 상담일지, 상벌점, 공지 (현장에서 앱 사용) |

### 3.2 회원가입 / 인증
- **학생**: 앱 설치 → 회원가입. 온보딩 4단계 = 회원가입 → OT(대면) → 학부모 인증(알림톡) → 정기일정 등록.
- **학부모**: 카카오톡 인증 가입, 학생 연락처 입력 → 자동 연동. 최대 2인.
- 🟡 확인필요: OT(대면) 단계가 앱 플로우를 막는 게이트인지, 학생-학부모 매핑 검증 로직.

### 3.3 출결 & 벌점
| 기능 | 상세 |
|---|---|
| 출결 키오스크 | NFC/RFID 카드 **또는** 앱 동적 QR(30~60초 유효) |
| 식사체크 키오스크 | 당일 급식 신청 여부 대조 후 수령 처리 |
| 벌점 자동 산출 | 결석·지각·조퇴·외출 자동 부여, 사유 승인 시 감면 |
| 조회 | 학생·학부모 일자별·월단위 |
- 흐름: 카드 태그/QR 스캔 → 키오스크가 서버로 **Webhook** → 출결 자동 처리 → 학부모 알림톡.

### 3.4 정기일정 등록
- 매월 1일 앱에서 현강·과외 등 외부 일정을 **월 단위** 등록 → 승인 → 벌점 미부여.
- 등록 시간 vs 실제 출입 시간 **30분 이상 차이 시 미인정**.
- 🆕 승인 주체: 3.15 라우팅 규칙에 따름(학부모/선생님 분리 대상).

### 3.5 사유 제출
- 결석·지각·조퇴·외출 발생 시 **당일 자정까지** 제출 → 즉시 승인 요청.
- **벌점 확정 후 사유 승인 불가** (벌점 확정 시점 정의 필요 🟡).
- 🆕 승인 주체: 3.15 라우팅 규칙에 따름.

### 3.6 사감 순찰 (스마트 순찰)
- 매 교시 자습실 **구역별 고정 QR** → 사감이 관리자 앱으로 스캔 → 순찰 시간 기록.
- 학생별 상태: 공부중 / 졸음 / 좌석에 없음.
- **교시 중 2회 연속 졸음 → 벌점 + 학부모 알림톡**. Daily Report 반영.

### 3.7 와이파이 방화벽 해제
| 항목 | 내용 |
|---|---|
| 해제 | 앱에서 시간(**최대 5시간**)·사유 입력 → 승인 |
| 자동 처리 | 승인 즉시 **Zyxel Nebula API 호출**로 해당 지점 해제, 시간 종료 시 자동 차단 |
| 다지점 | Nebula 클라우드 중앙 관제, 신규 지점 추가 시 개발 불필요 |
| 위반 | 해제 외 사이트 접속 벌점, 2회 적발 시 2주간 신청 제한 |
- 🔴 확인필요: Nebula API로 "특정 학생/특정 단말"만 해제 가능한지, 정책/그룹 단위인지 (제어 granularity).

## 5. 보안 설계 (ISMS-P 준용)

> 🆕 **7/10 책임 범위 정리 (중요):**
> **ISMS-P 인증 취득 자체는 개발팀(숀아이비엔엘) 소관이 아님.** 인증·정책 수립은 학원(내부)에서 검토·주관하며, **내부 검토 후 개발 반영이 필요한 보호대책만 요청** 주시면 개발 측에서 반영합니다. 즉 개발팀 책임 범위는 아래 "기술적 보호대책(개발 반영 항목)"에 한정. (이 내용을 별도로 정리해 대성/학원 측에 공유 예정)
